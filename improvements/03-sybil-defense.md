# 03 — Защита от Sybil: поручительство, распад R, вес × разнообразие графа

## Проблема

Исправленное Предложение о Sybil (после аудита) честно говорит: правило
$\sqrt{\R}$ **само по себе дробление не сдерживает, а поощряет**:

$$k\sqrt{r_0} \ \ge\ \sqrt{k\,r_0} \quad\text{(дробление выгодно)},\qquad
k = Q/\sqrt{r_0}\ \text{аккаунтов для }Q\text{ голосов}.$$

Против сообщества из $10^3$ участников со средней $r=100$
($\sum\sqrt r = 10^4$) нужно $>1.4\times10^4$ фейков при $r_0=0.5$ — много, но
вся защита держится на **цене одной личности** («KYC-лайт»). Внешний KYC —
покупаемый и обходимый; если цена личности упадёт до ~1 USD (фермы), атака
стоит ~\$14k. Нужны эндогенные барьеры, не зависящие от качества KYC-провайдера.

## Решение — три механизма (складываются)

### 1. Поручительство со слэшингом (identity cost становится эндогенной)

Вступление требует $m$ поручителей-действующих членов; каждый ставит на кон
$\sigma_R$ репутации (и/или $\sigma_V$ залога V) на испытательный срок $T_v$.
Доказанный фрод поручённого в течение $T_v$ → поручители теряют ставку.

- Цена фейковой личности для атакующего:
  $$c_{\mathrm{id}} \ \ge\ m\cdot\underbrace{\Pr[\mathrm{detect}]\cdot(\sigma_R^{\$}+\sigma_V)}_{\text{минимальная цена подкупа поручителя}} + c_{\mathrm{KYC}},$$
  т.е. атака = **коррумпирование реальных членов**, а не обман провайдера.
- Стоимость атаки на кворум: $k\cdot c_{\mathrm{id}} = (Q/\sqrt{r_0})\cdot c_{\mathrm{id}}$ —
  растёт с числом голосов и с надёжностью детекции.
- Побочный эффект: граф поручительств — готовый вход для мониторинга
  концентрации (фейковый кластер = плотный подграф поручительств).
- Параметры для старта: $m=2$, $\sigma_R = 5$ (≈ полгода активности),
  $\sigma_V = 50$ V, $T_v = 90$ дней.

### 2. Распад репутации при неактивности

$$r(t) = r_0\cdot 2^{-\Delta t_{\mathrm{inactive}}/T_{1/2}},\qquad
T_{1/2} = 180\ \text{дней (после 30 дней grace).}$$

- Sybil-эффект: поддержание $k$ аккаунтов требует **непрерывной** настоящей
  активности на каждом — постоянные операционные издержки $k\cdot c_a$/период,
  причём активность фейков между собой ловится мониторингом взаимности.
- Бонус для управления: «мёртвый вес» (ушедшие, но заслуженные) перестаёт
  голосовать; кворумы считаются по живому сообществу.
- Инварианты: I6 сохраняется (распад — не передача); в статье изменить фразу
  «R only minted and burned» → «minted, burned, and decays with inactivity».

### 3. Вес голоса × разнообразие графа (переиспользование антифрод-метрик)

Механизм уже считает двустороннюю концентрацию контрагентов. Использовать её
и в управлении:

$$\mathrm{vote}(m) = \sqrt{r_m}\cdot D_m,\qquad
D_m = \max\bigl(d_{\min},\ 1 - \mathrm{HHI}_m\bigr),$$

где $\mathrm{HHI}_m=\sum_j s_{mj}^2$ — Херфиндаль долей оборота члена $m$ по
контрагентам $j$, $d_{\min}=0.2$ — пол (новички не обнуляются).

Расчёт эффекта:

| Профиль | Контрагенты | HHI | $D$ | Эффективный голос |
|---|---|---|---|---|
| Честный активный | 10+ равномерно | ≈0.10 | 0.90 | $0.90\sqrt r$ |
| Честный новичок | 2–3 | ≈0.40 | 0.60 | $0.60\sqrt r$ |
| Фейк в кластере (пары) | 1–2 внутри кластера | 0.50–1.0 | 0.20 (пол) | $0.20\sqrt{r_0}$ |

Кластер получает **×0.2** к голосам → порог атаки из примера растёт с
$1.4\times10^4$ до $\approx 7\times10^4$ личностей (в 5 раз), либо атакующему
приходится торговать с настоящими членами (дорого, медленно, оставляет след).

- Инварианты: I7 сохраняется ($D$ — функция наблюдаемого состояния, не
  личности). Изменение правила голосования — проверить, параметрическое оно
  или конституционное; рекомендация: закрепить формулу конституционно.

## Совокупная стоимость атаки (пример)

Сообщество $10^3$ членов, $\bar r=100$; атака на репутационный кворум:

$$k \ \approx\ \frac{10^4}{\sqrt{r_0}\cdot D_{\mathrm{fake}}}
= \frac{10^4}{0.707\cdot 0.2} \approx 7.1\times10^4\ \text{личностей},$$

каждая: $m=2$ поручителя × слэшинг + непрерывная активность против распада +
KYC. И даже при успехе — второй (долевой) кворум покупается отдельно.
Атака дороже, чем построить конкурирующее сообщество.

## Изменения в коде симулятора

- `app/domain/services/voting.py`: множитель $D_m$ (данные HHI уже есть в
  `services/concentration.py`).
- `app/domain/services/reputation_delta.py` + периодическая операция
  `decay_reputation` (новая, по образцу `accrue_tenure`).
- Новая операция `vouch` + слэшинг в `process_overdue`/фрод-ремедиации;
  сущность `Vouch(voucher, vouchee, stake, expires)`.
- Эксперимент: сценарий `sybil_attack` — кластер из $k$ аккаунтов пытается
  провести голосование; критерий: доля голосов кластера < θ при $k$ вплоть до
  10× честного сообщества.

## Трейдоффы

- Поручительство замедляет рост (нужно найти 2 членов) — это фича для клуба,
  но ограничение для «вирального» роста; можно ослабить $m=1$ на раннем этапе.
- Распад R наказывает легитимные паузы (болезнь, отпуск) — grace 30 дней +
  «заморозка по заявлению».
- $D$-вес наказывает честные тесные пары (два партнёра, работающие только друг
  с другом) — пол $d_{\min}$ и учёт стажа смягчают.

## Статус

**Реализовано (Kredo v2).** D-вес голоса (`VotingService.diversity_weighted_power`
+ `ConcentrationMonitor`, параметр `vote_diversity_floor`); распад репутации
(`ReputationDecayService` + операция `DecayReputationOperation`, параметры
`reputation_half_life_days`/`reputation_decay_grace_days`, поле
`Member.last_active_tick`, вшита в периодический конвейер); поручительство
(`VouchingService` + `Vouch`/`VouchPolicy`). Эксперимент
`experiments/sybil_attack.py`.

Результаты (`results/sybil_attack.json`):
- **D-вес**: для большинства нужно 14 142 фейков (чистый √R) → **63 640** с
  D-весом (**×4.5**); честный D = 0.9, фейк-кластер D = 0.2.
- **Распад R (T½ = 180 д)**: сохраняется 56 % через 180 дней простоя, 27 % через
  год, 7 % через два — флот без активности тает.
- **Поручительство**: цена одной личности $55, полная атака на большинство —
  **$777 817**.

Распад — opt-in (`reputation_half_life_days=None` по умолчанию), D-вес касается
только управления (движок его не гоняет) — legacy-прогоны не затронуты
(та же статистика нарушений).
